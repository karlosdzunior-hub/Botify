## 1. Общее описание проекта

Bot Factory — это Telegram-бот, который позволяет любому пользователю создать собственного Telegram-бота без знания программирования.

**Основной флоу:** Пользователь описывает идею → ИИ задаёт вопросы → показывает план → генерирует код → проверяет ошибки → запускает бота на сервере.

### 1.1 Цели проекта
- Создать SaaS-платформу для автоматической генерации Telegram-ботов
- Монетизировать через систему кредитов + подписки
- Обеспечить автономную работу: генерация, проверка и запуск без участия оператора
- Выйти на окупаемость при 50+ платящих пользователях

### 1.2 Целевая аудитория
- **Малый бизнес** (кафе, салоны, магазины) — боты для записи и каталогов
- **Контент-мейкеры** — боты для рассылок и работы с аудиторией
- **Фрилансеры** — быстрое создание прототипов для клиентов
- **Агентства** — White Label решение под своим брендом

## 2. Функциональные требования

### 2.1 Главный интерфейс — решение противоречия бот vs мини-апп

**Основной интерфейс — Telegram Mini App** (раздел 2.4).
Обычный чат бота используется только для:
- `/start` — первое открытие и кнопка запуска мини-аппа
- Уведомлений (статус бота, пополнение кредитов, алерты)
- Команд управления без мини-аппа: `/mybots`, `/stop`, `/logs` (fallback)

Всё создание, диалог и управление ботами — только через мини-апп.

---

### 2.2 Пользовательский флоу создания бота

#### Шаг 1 — Приём идеи
- Пользователь вводит текстовое описание в чате мини-аппа
- Бот отвечает: «Принял, анализирую...»
- ИИ (Groq Llama) классифицирует тип: `simple` / `complex` / `miniapp`

#### Шаг 2 — Уточняющие вопросы
- ИИ задаёт 3–4 конкретных вопроса одним сообщением
- Пользователь отвечает свободным текстом
- Контекст хранится в Redis (FSM-состояние сессии)

#### Шаг 3 — Превью и подтверждение
- ИИ составляет план: функции, команды, сценарии
- Показывает расчётную стоимость в кредитах
- Кнопки: `[✅ Создать]` `[✏️ Изменить план]` `[❌ Отмена]`

#### Шаг 4 — Генерация кода
- Запрос токена бота у пользователя (через @BotFather)
- Валидация токена через Telegram Bot API (`getMe`)
- ИИ (Groq Qwen Coder) генерирует полный код на Python + aiogram 3
- Прогресс-бар обновляется через WebSocket в реальном времени

#### Шаг 5 — Авто-проверка и исправление ошибок
- Статический анализ: `ast.parse()` — синтаксическая проверка
- Запуск в Docker sandbox (`--network=none`, RAM 128MB, CPU 0.1)
- При ошибке — ИИ (Groq Qwen) автоматически исправляет
- До 3 итераций. Если не вышло — уведомление + возврат кредитов

#### Шаг 6 — Запуск и уведомление
- Бот запускается через **polling** (не webhook — см. п. 3.5)
- Пользователь получает ссылку на бота, статус, кнопки управления
- Кнопки: `[🔄 Перезапустить]` `[⏹ Остановить]` `[📋 Логи]` `[📥 Скачать код]`

---

### 2.3 Управление ботами (/mybots)
- Список всех ботов с иконкой статуса (✅/⛔/⚙️)
- Просмотр последних 20 строк логов в чате
- Удаление бота с подтверждением — останавливает процесс, удаляет файлы

### 2.4 Редактирование существующего бота
- Пользователь описывает изменение текстом
- ИИ патчит существующий код (diff-подход, не генерирует заново)
- Стоимость: 8 кредитов за добавление функции
- Бот автоматически перезапускается после изменений

### 2.6 Онбординг нового пользователя

1. `/start` → Telegram Bot отправляет приветственное сообщение + кнопку `[Открыть Bot Factory]`
2. Мини-апп открывается, показывает welcome-экран с 3 шагами: «Опиши бота → Подтверди план → Запусти»
3. Начисляется **30 приветственных кредитов** (`CREDITS_ON_REGISTER`)
4. Уведомление в чате: «🎁 Тебе начислено 30 кредитов. Создай первого бота бесплатно!»
5. Подсказка в поле ввода: «Например: бот для записи клиентов в кофейню»

### 3.2 API мини-аппа (FastAPI)

#### Аутентификация
Каждый запрос от мини-аппа должен содержать заголовок:
```
X-Telegram-Init-Data: <initData из Telegram.WebApp.initData>
```
Бэкенд валидирует подпись `initData` по алгоритму Telegram HMAC-SHA256.
При невалидной подписи — `401 Unauthorized`.

#### REST Endpoints

```
POST   /api/auth/validate          — Валидация initData, возврат JWT-сессии
GET    /api/user/profile           — Профиль, баланс, тариф
GET    /api/bots                   — Список ботов пользователя
POST   /api/bots                   — Создать нового бота (старт генерации)
GET    /api/bots/{bot_id}          — Детали бота
DELETE /api/bots/{bot_id}          — Удалить бота
POST   /api/bots/{bot_id}/start    — Запустить бота
POST   /api/bots/{bot_id}/stop     — Остановить бота
POST   /api/bots/{bot_id}/restart  — Перезапустить бота
GET    /api/bots/{bot_id}/logs     — Последние 20 строк логов
POST   /api/bots/{bot_id}/improve  — Добавить функцию / изменить бота
GET    /api/bots/{bot_id}/download — Скачать ZIP с кодом
POST   /api/token/validate         — Проверить токен @BotFather через Telegram API
GET    /api/credits/packages       — Список пакетов кредитов
POST   /api/credits/purchase       — Инициировать покупку кредитов
GET    /api/history                — История операций
GET    /api/referral               — Реферальная ссылка и статистика
POST   /api/hosting/purchase       — Купить хостинг для бота
```

#### Системные промпты (ключевые)

**Промпт классификации типа бота:**
```
Ты — классификатор Telegram-ботов. Определи тип бота по описанию пользователя.
Типы:
- simple: до 10 команд, без базы данных, без сложной логики
- complex: FSM, база данных, inline-кнопки, интеграции с API
- miniapp: нужен веб-интерфейс (React), сложная визуализация

Ответь ТОЛЬКО одним словом: simple, complex или miniapp.
```

**Промпт генерации кода:**
```
Ты — опытный Python-разработчик. Сгенерируй полный рабочий код Telegram-бота на Python + aiogram 3.x.

Требования:
1. Используй polling (не webhook)
2. Токен берётся из переменной окружения BOT_TOKEN
3. Логирование через logging в stdout
4. Обработка всех исключений
5. Код должен быть полным — без плейсхолдеров и TODO
6. Добавь requirements.txt

Описание бота: {description}
Функции: {features}
```

**Промпт проверки на вредоносность:**
```
Проверь описание бота на наличие запрещённого контента.
Запрещено: спам, фишинг, скам, сбор данных без согласия, нарушение ToS Telegram.
Ответь JSON: {"safe": true/false, "reason": "..."}
```

### 3.4 Схема отказоустойчивости
- Groq недоступен → автопереключение на Gemini Flash
- Gemini недоступен → повтор через Groq Qwen
- Claude недоступен → Gemini 2.0 Pro как фолбэк
- Повторные попытки с экспоненциальной задержкой: 1с → 2с → 4с
- Все переключения логируются в Sentry, пользователь не замечает сбоя

---

### 3.5 Режим работы сгенерированных ботов — Polling

**Решение:** все сгенерированные боты работают в режиме **long polling** (не webhook).

**Обоснование:**
- Не требует публичных URL и настройки Nginx для каждого бота
- Проще управлять через subprocess
- Достаточно для 99% use-cases

**Ограничения polling на бота:** `TELEGRAM_API_RATE_LIMIT = 30 req/sec`

---

### 3.6 Хранение и ротация логов

Каждый бот пишет логи в `generated_bots/user_{id}_{name}/logs.txt`.

**Ротация через logrotate:**
```
/app/generated_bots/*/logs.txt {
    size 5M
    rotate 3
    compress
    missingok
    notifempty
}
```
- Максимум 3 файла по 5MB на бота
- Старые логи сжимаются
- В API возвращаются последние 20 строк активного файла

## 5. Планировщик задач (Celery Beat)

Все задачи по расписанию выполняются через Celery Beat + Redis.

| Задача | Расписание | Описание |
|---|---|---|
| `expire_credits` | Каждый час | Сгорание кредитов с истёкшим `expires_at` |
| `notify_expiring_credits` | Каждые 6 часов | Уведомление за 7 дней до сгорания |
| `expire_hosting` | Каждый час | Остановка ботов с истёкшим хостингом |

## 7. Безопасность

### 7.1 Аутентификация мини-аппа
- Telegram передаёт `initData` при открытии мини-аппа
- Бэкенд валидирует HMAC-SHA256 подпись с `BOT_TOKEN` как ключом
- Выдаётся JWT-токен (TTL 24ч) для последующих API-запросов
- При невалидной подписи — `401 Unauthorized`, запрос не обрабатывается

### 7.2 Изоляция ботов
- Каждый бот — отдельный subprocess в своей директории
- `chmod 700` на директорию бота (доступ только владельцу процесса)
- Лимит ресурсов через `resource` (Python): 128MB RAM, ulimit CPU

### 7.3 Проверка кода (Sandbox)
- `ast.parse()` — синтаксическая проверка перед запуском
- Docker sandbox: `--network=none`, `--memory=128m`, `--cpus=0.1`
- Запрещённые паттерны: `os.system`, `subprocess`, `__import__`, `open('/etc')`
- ИИ-модерация запроса перед генерацией

### 7.4 Хранение секретов
- Токены ботов: зашифрованы Fernet AES-128
- Ключ шифрования: `FERNET_KEY` в `.env`
- API-ключи ИИ: только в `.env`, никогда в коде или логах
- Логи не содержат токенов (маскировка через `logging.Filter`)

## 12. Приём платежей

### 12.1 Способы оплаты

| Способ | Комиссия | Требования | Рекомендация |
|---|---|---|---|
| Telegram Stars | ~7% | Встроено в TG | ✅ Для старта |
| ЮКасса | 2.8–3.5% | ИП/ООО | ✅ Карты РФ |
| Robokassa | 3.5–5% | ИП/ООО | Альтернатива |
| Криптовалюта | 0–1% | — | Для продвинутых |

### 12.2 Флоу покупки кредитов
1. Пользователь нажимает `/buy` или открывает «Баланс» в мини-аппе
2. Видит список пакетов с кнопками
3. Выбирает пакет → Telegram показывает invoice (Stars) или редирект в ЮКассу
4. Оплачивает
5. Webhook подтверждает платёж (с проверкой идемпотентности)
6. Создаётся запись в `credit_batches` с `expires_at = now() + 90 days`
7. Обновляется `users.credits`
8. Пользователь получает уведомление с новым балансом

### 12.3 Идемпотентность платежей
- Каждый invoice имеет уникальный `payment_id`
- При получении webhook — проверка `payment_id` в таблице `transactions`
- Если уже обработан — `200 OK` без повторного начисления

## 14. Нефункциональные требования

### 14.1 Производительность
- Время ответа API: не более 500ms
- Время ответа бота на команду: не более 2 секунд
- Время генерации простого бота: 30–60 секунд
- Время генерации сложного бота: 60–120 секунд
- Одновременных генераций: до 10 (очередь для остальных)

### 14.2 Надёжность
- Uptime главного бота и API: 99.5%+
- Автоперезапуск при падении (supervisor watchdog)
- Ежедневный бэкап базы данных
- Все ошибки логируются в Sentry + алерт администратору

### 14.3 Масштабируемость
- Celery workers запускаются на отдельных машинах
- PostgreSQL выносится на отдельный сервер без изменения кода
- FastAPI API — горизонтальное масштабирование за Nginx
